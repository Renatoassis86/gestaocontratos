1. RESUMO EXECUTIVO APRIMORADO
--------------------------------------------------------------------------------
O ambiente Moodle da FICV não oferece apenas dados cadastrais e acadêmicos básicos. A documentação interna mostra que o token atual expõe uma camada ampla de dados em pelo menos seis níveis:

1. Dados mestres estruturados: usuários, cursos, categorias, matrículas, grupos, papéis, badges, eventos, notas, conclusão, calendários, blocos e arquivos.
2. Dados semiestruturados: customfields, preferences, roles, groups, enrolledcourses, courseformatoptions, warnings, advancedfeatures, plugins, fileareas, editorfields, contents, dates, completiondata.
3. Dados textuais não estruturados: descrições HTML de curso, textos de fóruns, blogs, tarefas discursivas, feedbacks docentes, conteúdo de workshops, campos de editor e mensagens.
4. Dados de arquivos e documentos: anexos, URLs de download, mimetype, tamanho, data de modificação, arquivos externos e áreas de arquivos.
5. Dados de engajamento e jornada: lastaccess, timeaccess, progress, hasprogress, isfavourite, overdue, completiondata, actionable, itens recentemente acessados.
6. Dados de governança técnica do próprio serviço: lista de funções habilitadas, capacidade de upload/download, quota, tamanho máximo de upload, sessões concorrentes, política aceita, homepage do usuário e chave privada de acesso a arquivos.

Conclusão executiva: o Moodle da FICV já está em um patamar em que a integração com seu sistema pode ir muito além de “cadastro + nota”. Dá para modelar uma base analítica institucional com dimensões de aluno, curso, professor, atividade, calendário, documentos, interações textuais, trilhas, badges, risco acadêmico e evidências documentais.

A documentação confirma que customfields não é um único campo, mas lista objetos de definição e valor. Permite criar ETL genérico que captura qualquer novo campo criado no Moodle FICV.

3.3. Tarefas discursivas com mais variáveis

Submissões trazem status de correção, anexos, estrutura por plugin, indicadores de atraso e bloqueio (`editorfields`de `text` e `format`). Excelente para Auditoria e Modelos de Risco.

3.4. Eventos e calendário trazem sinais analíticos prontos

 Flags como overdue, isactionevent, formattedtime. Permite criar Painel de Prazos e Monitor de Atrasos.

3.5. Conteúdo do curso tem estrutura de completion e datas

Dados de status (`completiondata`), datas de check e motivos de conclusão. Modelagem de trilha acadêmica progressiva.

3.6. blogs, fóruns e blocos ampliam a parte não estruturada

Blogs, config JSON de blocos, taxonomias e tags ampliando a mineração de texto.

4. NOVA TAXONOMIA RECOMENDADA PARA O RELATÓRIO
--------------------------------------------------------------------------------

A. Estruturado puro: IDs, nomes, datas, flags, números, status.
B. Estruturado relacional enriquecido: Papéis, grupos, matrículas, badges, calendários.
C. Semiestruturado aninhado: customfields, preferences, plugins, configs.
D. Não estruturado textual: summary, content, editorfields.text, posts, mensagens.
E. Não estruturado documental/binário: Arquivos, anexos, imagens, links de download.

5. RECOMENDAÇÕES DE ETL MAIS FORTES
--------------------------------------------------------------------------------

5.1. Arquitetura em 4 camadas:
- CAMADA 1: RAW INGESTION (JSON Bruto)
- CAMADA 2: STAGING NORMALIZADA (Explosão de Arrays e Semiestruturados)
- CAMADA 3: CORE RELACIONAL (users, courses, enrolments, calendar, files)
- CAMADA 4: ANALYTICS / MARTS (fact_engagement, fact_grade, fact_calendar, dim_user, dim_course)

5.2. Preservar HTML e Criar Texto Limpo: Armazenar `*_raw` e `*_plain_text`.
5.3. Estratégia Formal para Arquivos: Catálogo com hash/assinatura, mimetype, política de download.

6. RECOMENDAÇÕES NOVAS DE CIÊNCIA DE DADOS
--------------------------------------------------------------------------------

6.1. Modelos preditivos: Risco de Inadimplência acadêmica (Atas de submissão + ausências), Score de Engajamento Multimodal e Learning Analytics nativo.
6.2. NLP: Aplicar em posts, fóruns, respostas discursivas e feedback docente para análise de sentimento e clustering semântico.

7. GAPS QUE PRECISAM ENTRAR NO RELATÓRIO
--------------------------------------------------------------------------------

- Capacidade Massiva: Moodle 425 funções tem chamadas voltadas para UI, exigindo cuidado no bulk download.
- LGPD: Cuidado com dados sensíveis em feedbacks docentes e redações.
- Custo Computacional: Equilibrar Snapshots e Sync Reativo por Evento.

8. CONCLUSÃO APRIMORADA
--------------------------------------------------------------------------------

A documentação interna da API do Moodle da FICV demonstra que o ambiente expõe mais do que dados cadastrais tradicionais. A integração deve ser concebida em arquitetura multicamadas, contemplando tratamento de semiestruturas e governança documental. O ecossistema já oferece insumos suficientes para dashboards acadêmicos e analíticos avançados.

Abaixo está o mapeamento de variáveis de retorno para os endpoints fundamentais.

📘 PILAR 1: GESTÃO DE USUÁRIOS (User APIs)
--------------------------------------------------------------------------------
Endpoint Sugerido: `core_user_get_users_by_field` ou `core_user_get_users`

| Campo (Key) | Tipo | Descrição | Processo ETL / Tratamento |
| :--- | :--- | :--- | :--- |
| `id` | Int | ID sequencial único do usuário no Moodle FICV. | Converter para `moodle_id` (Chave Estrangeira no Supabase) |
| `username` | String | ID de Login (ex: CPF, Matrícula Institucional). | Sanitizar para lowercase para garantir unicidade. |
| `firstname` | String | Primeiro nome registrado. | Concatenar com lastname se o sistema destino usar Nome Completo. |
| `lastname` | String | Sobrenome registrado. | - |
| `email` | String | E-mail institucional ou pessoal. | Lowercase + Validação de Regex de e-mail p/ evitar falhas de webhook. |
| `idnumber` | String | Campo livre para matrícula externa. | Pode estar vazio. Usar CPF como backup alternativo. |
| `department` | String | Departamento ou ciclo. | Atributo categórico. |
| `customfields` | Array | **[SEMIESTRUTURADO]** Lista de campos personalizados. | **EXIGE LOOP**: Filtrar por shortname=`cpf` para extrair CPF limpo. |
| `preferences` | Array | Preferências do perfil do usuário. | Geralmente irrelevante para CRM, mas útil para logs de UI. |

📘 PILAR 2: GESTÃO DE CURSOS E CATEGORIAS (Course APIs)
--------------------------------------------------------------------------------
Endpoint Sugerido: `core_course_get_courses` ou `core_course_get_categories`

| Campo (Key) | Tipo | Descrição | Processo ETL / Tratamento |
| :--- | :--- | :--- | :--- |
| `id` | Int | ID sequencial único da disciplina/curso. | Chave primária de Curso no Supabase. |
| `fullname` | String | Nome completo da disciplina (ex: Ética Cristã). | Manter String original. |
| `shortname` | String | Nome curto (Cód. de Turma ou disciplina). | Útil para chaves de agrupamento (Tags). |
| `categoryid` | Int | FK da Categoria (ex: Graduação, Pós, etc). | Usar para fazer Join com `get_categories` no dashboard. |
| `summary` | HTML | Ementa ou descrição aberta do curso. | **NÃO ESTRUTURADO**: Fazer strip_tags() se quiser texto puro. |
| `startdate` | Unix | Timestamp de início. | **CASTING**: `new Date(val * 1000)` para ISO String. |
| `enddate` | Unix | Timestamp de fim. | **CASTING**: `new Date(val * 1000)` para ISO String. |
| `visible` | Int (0/1) | Status de publicação (Ativo/Oculto). | Map para boolean (true/false). |

📘 PILAR 3: DESEMPENHO ACADÊMICO (Gradebook APIs)
--------------------------------------------------------------------------------
Endpoint Sugerido: `gradereport_user_get_grade_items`

| Campo (Key) | Tipo | Descrição | Processo ETL / Tratamento |
| :--- | :--- | :--- | :--- |
| `userid` | Int | ID do aluno. | Linkar com tabela central de alunos. |
| `courseid` | Int | ID do curso associado à nota. | - |
| `itemname` | String | Nome da atividade (ex: Quiz 1, Prova Final). | Filtrar p/ excluir 'Category Total' se quiser médias brutas. |
| `itemtype` | String | 'course', 'category' ou 'mod' (Atividade). | Se for 'course', é a média final da disciplina. |
| `graderaw` | Float | Avaliação numérica bruta. | Calcular `(graderaw / grademax) * 10` se quiser normalizar escala 0-10. |
| `grademax` | Float | Teto de pontuação do item. | Geralmente 100.0 ou 10.0. |
| `percentage`| String | Nota em formato percentual (ex: "85.00 %"). | Converter para float (substituir '%' por vazio). |

📘 PILAR 4: MATRÍCULAS E GRUPOS (Enrolment APIs)
--------------------------------------------------------------------------------
Endpoint Sugerido: `core_enrol_get_enrolled_users` ou `core_enrol_get_users_courses`

| Campo (Key) | Tipo | Descrição | Processo ETL / Tratamento |
| :--- | :--- | :--- | :--- |
| `id` | Int | ID do usuário. | FK com central de alunos. |
| `roles` | Array | Lista de cargos (Student, Teacher, Manager). | **DIAGNÓSTICO**: Fundamental para identificar o "Professor". |
| `roles.shortname` | String | Cód do cargo (ex: 'student', 'editingteacher').| Filtrar 'editingteacher' para extrair Docente Atribuído. |



================================================================================
          MELHOR ARQUITETURA DE INTEGRAÇÃO INDICADA                             
================================================================================

Com base nas orientações de WebServices do Moodle, a melhor arquitetura para o seu sistema é:

**🏆 ARQUITETURA HÍBRIDA (Pull-based via REST + Push-Events)**

1. **Sincronização Periódica Incremental (REST API - Pull)**
   - **Indicação**: Coleta de dados volumosos (Notas de turmas inteiras, Atualização cadastral).
   - **Como funciona**: Loops de lotes (Pagination) usando `core_enrol_get_enrolled_users` 
     puxando novos dados a cada 1 hora para inserir com trigger UPSERT no Supabase.
   - **Vantagem**: Menos acoplado, tolerante a falhas rápidas na conexão, fácil de debugar.

2. **Gatilhos Reativos (Event Monitoring / Webhooks - Push)**
   - **Indicação**: Disparos instantâneos (ex: Aluno concluiu o Curso -> Gerar Contrato/Certificado).
   - **Como funciona**: Configurar o Moodle FICV para disparar requisição POST p/ webhook `/api/moodle/completed` 
     do seu sistema destino no evento `course_completed`.
   - **Vantagem**: Reduz o tráfego de pooling de APIs e entrega tempo-real em operações críticas.

### Resumo do Próximo Passo do Desenvolvedor:

usando Paginação via loops de ID de curso para evitar estouro de memória e rate-limit.
